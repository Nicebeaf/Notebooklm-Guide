# 第 10 章 从陌生到理解：建立行业地图与关键问题

> 贯穿案例：AI Agent 行业
> 本章全部功能状态与数据核验日期：2026 年 7 月 10 日。凡正文出现的数字，作者均已打开原始出处页面核对；无法核验的数字一律不出现。

假设你下周要见一个 AI Agent 方向的创业者，或者你的老板让你评估「我们要不要做 Agent 产品」，又或者你只是想写一篇不人云亦云的行业文章。你对这个行业的全部认知来自刷到的十几条短视频和几篇公众号文章：这就是「陌生」。

本章要解决的问题是：如何用 NotebookLM，在一到两个工作日内，从陌生走到「理解」。注意交付物的边界：本章交付的是**认知地图**，你知道这个行业分几层、关键数字的口径、哪些是已确认的事实、哪些只是趋势或假设、哪里还没搞清楚。它不回答「该不该投资」「该不该入场」，那是决策报告的事，属于第 12 章。

## 一、先想问题，再囤资料

大多数人研究一个行业的第一步是搜索和收藏。这个顺序是错的。没有问题的资料搜集，结果只是把信息焦虑从「不知道」变成「囤了五十篇没读」。

第一步永远是写下研究问题清单。以 AI Agent 行业为例，我实际使用的清单是：

**主问题**：AI Agent 行业在 2026 年处于什么阶段？哪些场景已经出现付费与规模化信号？什么被高估了？

**子问题**：

1. 企业落地的真实渗透率是多少？「实验」「单职能规模化」「多职能规模化」必须分开统计。
2. 技术栈分几层？MCP、A2A 这些协议各解决什么问题？标准由谁治理？
3. 哪些应用场景已有付费与规模化的公开信号，证据强度如何？
4. 市场规模有几种统计口径？哪些口径的方法论是公开的？
5. 玩家格局：模型厂商、云厂商、应用层创业公司各占哪一层？
6. 目前最主要的落地失败原因是什么？

注意主问题的措辞。第一稿写的是「谁在赚钱」，研究结束时发现根本答不了：公开来源里几乎没有可靠的分厂商收入数据。一个无法用可获得证据回答的研究问题，要么改问题，要么补证据。我选择了改问题：「哪些场景出现付费与规模化信号」是同一个关切的可回答版本。**研究问题必须与证据的可获得性匹配，这本身就是研究设计的一部分。**

同时写下交付标准：一份产业链分层图、一张关键数字表（每个数字带口径、出处和局限）、一份按证据强度分级的地图。有了交付标准，你才知道什么时候可以停。

搜集顺序（第 2 章原则的落地）：**定义问题 → Fast Research 初筛候选 → 人工逐项审核 → 对证据不足的子问题启动 Deep Research → 审核后导入**。Deep Research 是补枪，不是起手式。原因见本章失败案例。

## 三、建立 Notebook

命名：`行研-AI Agent-2026Q3`（规范见第 9 章）。导入时注意第 6 章的限制：网页只进 HTML 文本；报告尽量找原始 PDF 而非转载页；门控报告（如来源 5）导入的只是落地页，正文进不来，这正是它的数字不能引用的原因。

导入后先做一件事：给每个来源打层级标签（一手/综述/媒体），并在 Notebook 自定义指令里写明：「回答时优先引用一手与综述来源；引用媒体来源时必须注明；来源之间数字冲突时并列呈现，不要调和。」

## 四、提示链：六步从来源到地图

以下提示链按顺序执行，每一步的输出是下一步的输入。更多可直接复制的提示语见附录 D。

**P1 · 分层**：「基于全部来源，列出这个行业的产业链分层，每层给出代表公司或项目，每个判断注明来源。」
**P2 · 口径**：「列出所有来源中出现的市场规模与落地率数字：数值、统计口径、测量对象、样本、发布机构、发布日期。指出互相矛盾之处，不要调和它们。」
**P3 · 渗透率**：「关于企业落地率，各来源的数字和统计定义分别是什么？哪些统计『实验』，哪些统计『规模化』？规模化是按组织算还是按职能算？」
**P4 · 争议**：「找出来源之间观点直接冲突、或证据强度明显不足的三个问题，分别列出论据和出处。」
**P5 · 缺口**：「对照我的研究问题清单（粘贴清单），哪些子问题在现有来源中证据不足？」，证据不足的子问题，才是启动 Deep Research 定向补源的对象。
**P6 · 地图**：「生成行业认知地图大纲，按四级证据强度组织：已确认事实、多来源支持的趋势、工作假设、待验证问题。每条注明来源。」

P6 之后生成可视化 Mind Map 时注意一个容易做错的操作细节：**聊天回答不会自动成为 Mind Map 的输入**，官方说明中，Mind Map 总结的是选中的来源，不是对话。正确步骤是：将 P6 的回答 **Save to Note** → 必要时人工整理 → **Convert to source**（Note 可转为来源，官方支持，单条或全部皆可）→ 在来源面板只勾选这个新来源 → 生成 Mind Map。这样图谱反映的是你核验过的地图，而不是全部原始来源的自动摘要。

P2、P3、P4 的共同点是**逼模型暴露矛盾**。「不要调和它们」是整条提示链里最重要的半句话。

## 五、关键数字表：口径核验的现场教学

这张表是本章的核心交付物。它展示的不是「行业数据」，而是**每个数字被允许印出来之前要经过什么**：

| 指标 | 数字 | 测量对象 | 阶段定义 | 样本 | 调查/发布日期 | 原始出处 | 局限 |
|---|---:|---|---|---|---|---|---|
| 组织规模化比例 | 23% | 受访组织 | 至少一个职能在规模化 agentic AI 系统 | 1,993 人 / 105 国 | 调查 2025-06-25～07-29，发布 2025-11-05 | McKinsey State of AI 2025（来源 1） | 自报数据；受访者构成偏大型组织 |
| 组织实验比例 | 39% | 受访组织 | 已开始实验 AI Agent | 同上 | 同上 | 同上 | 同上 |
| 单一职能内规模化 | ≤10% | 任一给定职能 | 该职能内在规模化 | 同上 | 同上 | 同上；经 Forbes 2026-03-22 报道交叉核对（来源 9） | 多数规模化组织只覆盖一两个职能 |
| A2A 支持组织数 | 150+ | 协议生态 | 公开表态支持 | 公告口径 | 2026-04-09 | Linux Foundation 新闻稿（来源 2） | 项目主办方自报；「支持」定义宽泛 |
| A2A 代码库关注度 | 22,000+ stars | GitHub 仓库 | 星标数 | — | 同上 | 同上 | 星标≠使用 |
| MCP 生态规模 | 月 SDK 下载 9,700 万+；活跃服务器 10,000+ | 协议生态 | SDK 下载量与服务器数 | 官方口径 | 2025-12-09 | MCP 官方博客（来源 4） | 项目方自报；下载量≠活跃使用 |

读这张表的方法：前三行来自**同一份调查**，却讲出三个不同故事，「23% 在规模化」听起来行业已经起飞；换成职能视角，任何一个具体职能内规模化的都不超过 10%。同一份数据，按组织算和按职能算，热度差一倍以上。**口径核验的第一课不是对比不同来源，而是把同一来源内部的统计定义读完整。**

这张表还有一个刻意的空缺：**中国市场规模**。智研咨询的公开大纲只披露了较早年份的历史数据，2025 年最新规模及其测算方法都在付费报告内，本章未能核验，因此不引用任何具体数字。知道什么时候不写数字，也是一种研究能力，一个空缺加一句说明，比一个无法追溯的数字诚实得多。

事实与推断的分界也在这里：「23% 的组织至少在一个职能规模化」是事实；「行业处于从实验向单职能规模化过渡的阶段」是我基于事实的**推断**，在地图中用不同标记区分。

## 六、失败案例：这一章自己是怎么跑砸的（两次）

**第一次失败：批量导入污染。** 第一轮我按多数人的直觉做法，搜「AI Agent 2026 statistics」并把结果前二十条批量导入。前二十条里近一半是内容农场式 SEO 页（《200+ Data Points》《120+ Enterprise Data Points》之类），互相转抄同一批数字、不注明原始出处、多口径混排。导入后，某个未经证实的渗透率数字在回答里成了「多来源证实」的定论，实际是同一数字被转抄了五六次。**来源数量制造虚假置信度。** 修正：删库重来，按本章流程先写问题清单再人工选源。

**第二次失败：转述数字直接进正文。** 更隐蔽，也更难堪：本章初稿自己也栽了。初稿的「冲突现场」一节并列了三组落地率数字，看起来完成了口径对比，直到逐个回溯原始出处时发现：

| 初稿引用 | 回溯结果 | 处理 |
|---|---|---|
| 「Google Cloud 调研：70% 企业已在生产环境运行 Agent」 | 报告为门控下载，公开页面无此数字；二手转述中另有 52% 的说法，同样无法定位原文 | **删除** |
| 「S&P Global：31% 的组织有 Agent 在生产环境」 | S&P 公开文章只说「超过半数处于生产或 PoC 阶段」，31% 无法定位到调查名称与图表 | **删除** |
| 「MCP 月下载量过亿」 | 追溯链条终点是内容农场，正是第一次失败中被剔除的那类站点 | **删除**；后在 MCP 官方公告中找到可核验口径（月 SDK 下载 9,700 万+，2025-12-09），按新出处重新收录 |

三个数字全部来自研究过程中的**搜索摘要和二手汇编**，初稿把「有引用格式」误当成了「已核验」。这正是第 7 章警告的引用链断裂：引用存在，但链条中断在转述层，到不了原始出处。修正规则由此写入本书验收清单：**凡要印出的数字，作者必须亲手打开原始出处页面；打不开、找不到、需要付费墙后才能确认的，一律不印。**

这两次失败共同证明了流程顺序的必要性：「让 Agent 搜得越多，研究就越好」是错的；「有引用就等于可靠」也是错的。

## 七、最终成品：AI Agent 行业认知地图（2026 年 7 月版）

按证据强度分四级（Mind Map 可视化版与完整数字表随线上资料包提供）：

**本章采用的分析框架**（先于四级分类声明）
为便于理解，本章将 AI Agent 产业划分为：模型层（Anthropic、OpenAI、Google 等）→ 协议与基础设施层 → 平台层（云厂商 Agent 平台）→ 应用层（垂直 Agent）。这是作者综合来源后采用的**工作框架**，不是某份来源直接确认的事实，也不代表唯一的行业分类法，不同报告可能分三层、四层或六层。分析框架和事实必须分开声明。

**A. 已确认事实**（一手出处，作者核验）
- MCP 规范现行版本为 2025-11-25 版（来源 3）；2025 年 12 月，Anthropic 将 MCP 捐赠给 Linux Foundation 旗下新成立的 Agentic AI Foundation，MCP 为创始项目之一，月 SDK 下载量 9,700 万+（来源 4）。
- A2A v1.0 于 2026-04-09 公告发布：150+ 支持组织，SDK 覆盖 5 种语言，已集成入 Google、Microsoft（Azure AI Foundry、Copilot Studio）、AWS（Bedrock AgentCore）平台；官方定位与 MCP 互补，A2A 管 Agent 间协作，MCP 管 Agent 连接工具与数据（来源 2 原文）。
- A2A 路线图包含互操作规范、注册表整合与安全最佳实践（来源 2）；Agent 支付协议 AP2 已有 60+ 组织支持（来源 2）。
- 企业落地：23% 组织至少一职能规模化、39% 实验中、任一单一职能内规模化 ≤10%（来源 1，见数字表）。

**B. 多来源支持的趋势**（方向一致，但缺独立收入验证）
- 多份材料把 IT、知识管理、客服、编码等列为较早落地的场景；其中麦肯锡调查明确指出，Agent 使用在 IT 和知识管理职能中最常见（来源 1）。
- 企业整体处于「实验多、单职能规模化少、跨职能规模化更少」的过渡带。

**C. 作者工作假设**（明确标注，待证据升级）
- MCP＋A2A 的组合可能成为企业多 Agent 系统的常见架构（官方仅说「互补」，「默认架构」是我的外推）。
- 行业叙事正从「工具」向「数字员工」迁移（叙事观察，无量化支持）。
- 商业模式将多轨并行（MaaS/PaaS/SaaS/RaaS 分类来自中文研究机构摘要，方法论未公开）。

**D. 待验证问题**（直接成为第 12 章的问题清单）
- 各场景的真实付费与收入数据（现有来源几乎全是成功叙事与自报调研）。
- Agent 项目的失败率与失败归因。
- 中国市场规模：公开大纲只披露较早历史数据，最新规模及测算方法位于付费报告内，本章未能核验，因此不引用具体数字。
- MCP 与 A2A 各自的规范演进、治理机制，以及二者在实际系统中的组合方式。

从陌生到这张地图，实际用时：来源搜集与审核约半天，提示链、核验与返工约一天，返工的一天就是第六节那两次失败的代价，也是这一章最值钱的部分。

## 八、本章验收清单（全书统一模板）

- ☐ 研究问题清单先于任何来源搜集写出，且每个问题与证据可获得性匹配
- ☐ 来源包三层齐备，每项含完整链接、发布日期与定位信息；内容农场类已剔除
- ☐ **凡印出的数字，作者本人能打开、能核对、能精确定位原始出处**；无法做到的一律不印。已合法访问的付费来源可以引用，但须注明访问限制，且不得分发原文件
- ☐ 同一来源内部的口径差异已读完整（组织口径 vs 职能口径这类）
- ☐ 冲突数字并列呈现并解释口径，未取平均
- ☐ 地图按四级证据强度组织，事实与推断分离
- ☐ 待验证问题可直接作为下一个 Notebook 的问题清单
- ☐ 全章功能状态与数据标注核验日期（本章：2026-07-10）
